# s-shoter — Техническое задание (MVP)

> Документ для исполнителя — AI-агента (Claude Code). Реализуй строго в рамках scope MVP. Всё, что помечено `[POST-MVP]`, не делать в первой версии.

## 1. Краткое описание

**s-shoter** — веб-приложение для сохранения скриншотов на бесконечную доску (как Miro) вместе с полным контекстом, в котором скриншот был сделан. Состоит из трёх частей:

1. **Chrome-расширение (MV3)** — делает скриншот текущей вкладки одним кликом, собирает контекст (URL, заголовок, время) и отправляет на бэкенд. В попапе показывает все скриншоты за сегодня (количество и время).
2. **Веб-приложение** — бесконечная доска (canvas) со всеми скриншотами пользователя в виде карточек. Можно панорамировать/зумить, открыть скриншот и посмотреть детали.
3. **Бэкенд + БД** — регистрация/авторизация, хранение скриншотов (файл + метаданные), API для расширения и веб-клиента.

**Ключевая ценность:** скриншот никогда не теряет контекст — рядом всегда есть URL, заголовок страницы и время, и всё это лежит на одной визуальной доске.

## 2. Цель MVP и критерии готовности

MVP считается готовым, когда выполнимы все сценарии:

1. Пользователь регистрируется по email + паролю и входит в веб-приложение.
2. Пользователь устанавливает расширение, логинится в нём (тем же аккаунтом) и нажатием кнопки сохраняет скриншот текущей вкладки.
3. Скриншот с контекстом (URL, title, timestamp) появляется на бесконечной доске в веб-приложении.
4. На доске можно панорамировать и зумить; карточки скриншотов перетаскиваются, позиция сохраняется.
5. Клик по карточке открывает детали скриншота (полное изображение + все метаданные).
6. Попап расширения показывает скриншоты за сегодня: их количество и время каждого.

## 3. Вне scope (НЕ делать в MVP) `[POST-MVP]`

Командная работа и шаринг досок; комментарии/аннотации поверх скриншотов; теги, папки, поиск; несколько досок у одного пользователя; OAuth/соцсети; выделение области экрана (скриншот всегда = видимая часть вкладки); мобильная версия; экспорт; OCR по скриншотам.

## 4. Технологический стек (предложение)

Стек подобран под быструю и надёжную сборку одним агентом, с минимумом инфраструктуры.

**Монорепозиторий** (pnpm workspaces):

```
s-shoter/
  apps/
    web/         # Next.js — UI + API
    extension/   # Chrome MV3
  packages/
    db/          # Prisma schema + client
    shared/      # общие типы (zod-схемы DTO)
```

**Веб + API — Next.js 15 (App Router, TypeScript).** Один проект отдаёт и UI, и REST API (route handlers). Меньше движущихся частей, чем отдельный фронт + бэк.

**UI — shadcn/ui (строго и только он) + Tailwind CSS v4.** Все компоненты интерфейса (формы, кнопки, поля, модалки, панели, тосты) — примитивы shadcn/ui, добавляются через **shadcn MCP** из реестра `@shadcn`. Руками не писать компонент, который есть в реестре. Стилизация — только семантические токены (`bg-background`, `text-muted-foreground`, `rounded-md` и т.д.), без hex/rgb/произвольных значений в квадратных скобках, без CSS-in-JS и инлайн-`style`. **Все правила дизайн-системы и структуры кода — в `AGENTS.md` (см. §4.1), они обязательны на каждом файле.** Расширение тоже использует Tailwind v4 + те же токены (shadcn-компоненты по возможности).

**Бесконечная доска — `react-konva` (Konva.js).** Canvas с pan/zoom и перетаскиваемыми карточками. Полный контроль над рендером, лёгкий для MVP. *Альтернатива — `tldraw`, если нужен готовый Miro-подобный движок «из коробки»; но он опинионатед и тяжелее кастомизировать, поэтому для MVP берём `react-konva`.*

**Важный нюанс для доски:** Konva рендерит в `<canvas>`, а не в DOM, поэтому Tailwind-классы и токены к фигурам неприменимы — цвета задаются строками в JS. Правило «никаких hex в компонентах» здесь соблюдается так: завести **один модуль** (`apps/web/lib/canvas-theme.ts`), который читает значения CSS-токенов (через `getComputedStyle` по `--background`, `--border`, `--primary` и т.д.) и отдаёт их Konva. Никаких хардкод-цветов, разбросанных по canvas-коду. Сам DOM-чром доски (тулбар, кнопки зума, панель деталей) — обычные shadcn-компоненты с токенами.

**База данных — SQLite + Prisma ORM.** Никакого Docker и серверов БД: база — это локальный файл (`packages/db/dev.db`), Prisma работает с ним через `provider = "sqlite"`. Достаточно для MVP и однопользовательской разработки. *Замена на PostgreSQL позже — смена одной строки в схеме Prisma + строки подключения.*

**Авторизация — Auth.js (NextAuth v5), провайдер Credentials (email+пароль).** Пароли — `bcrypt`. Сессии — JWT. Расширение авторизуется тем же бэкендом (см. §8.1).

**Хранение картинок — локальная папка на диске.** Файлы складываются в каталог `apps/web/storage/` (вне git, в `.gitignore`). В БД хранится относительный путь/имя файла и метаданные, не сам файл. Картинки отдаются клиенту через защищённый route бэкенда (`GET /api/files/:key`), который проверяет, что файл принадлежит текущему пользователю, — прямого доступа к папке снаружи нет. *Замена на S3/R2 позже — за интерфейсом storage-слоя.*

**Расширение — Chrome Manifest V3**, vanilla TS + Vite (без тяжёлого фреймворка). API `chrome.tabs.captureVisibleTab` для снимка.

**Валидация DTO — zod**, общие схемы в `packages/shared`, используются и на бэке, и на фронте, и в расширении.

**Качество:** ESLint + Prettier + TypeScript strict. Тесты — Vitest (юнит) + минимум 1 e2e-сценарий регистрации (Playwright) `[опционально, если успеваешь]`.

> Если у тебя есть основания выбрать другой стек — сначала предложи и обоснуй, не меняй молча (см. поведенческие правила в начале).

### 4.1. Конвенции кода и дизайн-системы (обязательно)

В корне репозитория лежит **`AGENTS.md`** — единый источник правил по стилям и структуре кода. Эти правила обязательны на каждом файле фронтенда, это не рекомендации. Кратко:

- Только семантические токены shadcn — никаких hex/rgb/hsl и произвольных Tailwind-значений `[...]` для цвета, отступов, радиуса, шрифта. Нет токена — добавь именованный токен в тему, не выдумывай инлайн.
- Сначала спроектируй общие компоненты (`components/`), потом собирай из них страницы. Общий «хром» (header/nav/контейнер) живёт в `layout.tsx`, а не в страницах. Не копипастить блоки между страницами.
- shadcn-примитивы добавляются через shadcn MCP из реестра `@shadcn`. Реестр не угадывать.
- Повторяющиеся строки, навигация, списки роутов — в `config/`.
- Задача не завершена, пока не прошёл lint без ошибок и пока в диффе компонентов нет `#`, `rgb(`, `-[`.

## 5. Модель данных

Prisma-схема (ориентир, поправь типы под себя):

```prisma
model User {
  id           String       @id @default(cuid())
  email        String       @unique
  passwordHash String
  createdAt    DateTime     @default(now())
  screenshots  Screenshot[]
}

model Screenshot {
  id          String   @id @default(cuid())
  userId      String
  user        User     @relation(fields: [userId], references: [id], onDelete: Cascade)

  // файл
  storageKey  String   // имя файла в apps/web/storage/
  width       Int
  height      Int
  mimeType    String   @default("image/png")

  // контекст
  sourceUrl   String   // URL страницы
  pageTitle   String   // <title> вкладки
  capturedAt  DateTime // время снимка (с клиента)

  // позиция на доске
  boardX      Float    @default(0)
  boardY      Float    @default(0)
  boardScale  Float    @default(1)

  createdAt   DateTime @default(now())

  @@index([userId, capturedAt])
}
```

## 6. Экраны веб-приложения

**6.1. `/register`** — форма: email, пароль, подтверждение пароля. Валидация (email формат, пароль ≥ 8 символов). При успехе — автологин и редирект на доску. Ошибки (email занят, слабый пароль) показываются под полями.

**6.2. `/login`** — email + пароль. При успехе → доска. Ссылка на регистрацию.

**6.3. `/board`** (главный экран, требует авторизации) — бесконечная доска:
- Canvas на весь экран, фон в стиле «space» (тёмный, точечная/grid-сетка).
- Скриншоты — карточки-превью на своих координатах (`boardX/Y/scale`).
- Жесты: pan (drag по пустому месту или пробел+drag), zoom (колесо / pinch).
- Перетаскивание карточки → debounce 500 мс → PATCH позиции на сервер.
- Клик по карточке → панель/модалка деталей (6.4).
- Пустое состояние: подсказка «Установи расширение и сделай первый скриншот».

**6.4. Детали скриншота** (панель сбоку или модалка) — полное изображение, URL (кликабельный, открывается в новой вкладке), заголовок страницы, время снимка, кнопка «Удалить».

## 7. Chrome-расширение

**7.1. Попап** (`popup.html`):
- Если не залогинен → форма входа (email+пароль) или кнопка «Войти», ведущая на `/login` веб-приложения.
- Если залогинен → большая кнопка **«Сохранить скриншот»** + список скриншотов **за сегодня**: для каждого — мини-превью/иконка и время; сверху счётчик «Сегодня: N».

**7.2. Логика снимка:**
1. По кнопке — `chrome.tabs.captureVisibleTab` → PNG dataURL.
2. Собрать контекст активной вкладки: `tab.url`, `tab.title`, `capturedAt = new Date()`.
3. Загрузить на бэкенд (см. §8.3). Показать результат (успех/ошибка) в попапе.
4. Обновить список «за сегодня».

**7.3. Хранение токена:** access-токен — в `chrome.storage.local`. «Сегодня» = с локальной полуночи пользователя.

## 8. API

База: `/api`. Все ответы — JSON. Валидация тел — zod-схемами из `packages/shared`. Ошибки — `{ error: { code, message } }` + корректный HTTP-статус.

**8.1. Авторизация.** Веб использует сессии Auth.js (cookie). Расширение не может полагаться на cookie сторонней вкладки, поэтому для него — Bearer-токен:
- `POST /api/auth/register` → `{ email, password }` → создаёт юзера, ставит сессию.
- `POST /api/auth/login` (для расширения) → `{ email, password }` → `{ accessToken }` (JWT, срок ~30 дней).
- Эндпоинты скриншотов принимают **либо** сессионную cookie (веб), **либо** `Authorization: Bearer <token>` (расширение).

**8.2. Скриншоты — чтение/доска:**
- `GET /api/screenshots` → список скриншотов текущего юзера: метаданные, позиция и URL картинки (вида `/api/files/:key`).
- `GET /api/screenshots/:id` → детали + URL полного изображения.
- `PATCH /api/screenshots/:id` → `{ boardX, boardY, boardScale }` — сохранить позицию.
- `DELETE /api/screenshots/:id` → удалить запись и файл из папки `storage/`.
- `GET /api/files/:key` → отдаёт сам файл картинки; проверяет, что скриншот с этим ключом принадлежит текущему юзеру (иначе 403/404).

**8.3. Загрузка скриншота** (из расширения):
- `POST /api/screenshots` — `multipart/form-data`: файл изображения + поля `sourceUrl`, `pageTitle`, `capturedAt`. Сервер сохраняет файл в папку `storage/`, создаёт запись, возвращает созданный объект. Новые скриншоты получают стартовую позицию каскадом, чтобы не накладывались друг на друга.

**8.4. «За сегодня»** (для попапа):
- `GET /api/screenshots/today` → скриншоты с `capturedAt` за текущие сутки (учёт таймзоны клиента через query-параметр `tz` или смещение) → `{ count, items: [{ id, capturedAt, thumbnailUrl }] }`.

Все эндпоинты строго фильтруют по `userId` из сессии/токена — пользователь видит только свои скриншоты.

## 9. Нефункциональные требования

- TypeScript strict, без `any` в публичных интерфейсах.
- Секреты — через `.env` (есть `.env.example` со всеми ключами: `DATABASE_URL="file:./dev.db"`, `AUTH_SECRET`). Папка `storage/` и `*.db` — в `.gitignore`.
- Лимит размера загружаемого файла (например, 10 МБ), валидация mime (только png/jpeg).
- Пароли только хэшированные, токены не логируются.
- README с инструкцией локального запуска (без Docker): `pnpm install` → `pnpm db:migrate` → `pnpm dev` → как загрузить расширение в Chrome (`chrome://extensions`, Load unpacked).
